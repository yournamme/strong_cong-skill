# Project Dev Workflow

面向的场景:用户是电子信息工程背景,做比赛/课程项目,习惯"先和AI讨论方案,再让AI拆成任务,然后AI一步步执行、用户负责关键节点验证"。这个技能把整个链路管起来,核心设计原则是:**单一文档承担计划、进度、历史三重角色**,避免多份文档互相不同步。

整个流程分三个阶段,按需进入,不用每次都从头走一遍——如果用户已经有明确方案,直接从阶段二开始。

---

## 阶段一:方案讨论(仅当想法还比较模糊/宽泛时使用)

目标不是帮用户畅想更多可能性,而是尽快把想法收敛成一个具体、能落地、匹配用户时间/资源限制的最小可行方案(MVP)。

**讨论纪律(重要):**

1. 每一轮最多提出 3~5 个关键问题,一次性批量问完,不要一个问题一个问题来回问。
2. 只问真正决定方案方向/可行性的问题,不要纠结无关紧要的细节。
3. 如果用户的回答已经足够合理推断出答案(哪怕不是100%确定),不要揪着同一个点反复追问——直接给出假设并标注"这是我的假设,如有不对请指正",让用户确认/纠正就行,不要陷入无限澄清循环。
4. 每一轮结束时,先总结"已经明确的部分",再列"还需要确认的关键问题",让讨论有推进感。
5. 全程保持"够用就好"的克制:一旦讨论中冒出来的功能点明显超出用户的时间/资源限制,主动提示"这个可能超出MVP范围,建议放到以后再做的清单里"。
6. **收敛条件**:当下面 1~6 项(涉及硬件再加 7~8 项)都已经明确到"可以直接拿去写任务拆解文档"的具体程度时,停止提问,输出最终方案摘要。

**需要收敛到的信息:**

1. 一句话项目目标
2. 目标用户是谁
3. 3个典型使用场景
4. MVP 必须做的功能(最小可用,砍到不能再砍)
5. 可以以后再做的功能
6. 成功标准(具体、可判断,不要写"效果好"这种)
7. (涉及硬件/比赛)硬性约束:截止时间、评分规则里的硬指标、必须用到的器材/平台
8. (涉及硬件/比赛)关键技术选型假设

产出一份"项目方案摘要",作为阶段二的输入。

---

## 阶段二:任务拆解

### 第一步:先判断项目规模,不要默认深拆

判断依据:模块/组件数量、代码量级、是否涉及硬件/实机、技术不确定性、AI agent 能否连续独立完成大部分工作。拿不准时优先选粗粒度,不要为了"看起来专业"而过度拆分。

- **模式A(简单/可一次性完成)**:单一脚本、逻辑清晰、AI能一口气写完调通、内部没有高风险点。此时不要拆成"任务1/2/3",只需要一份整体完成标准 + (如有)1~2个检查点。
- **模式B(中大型/多阶段/含硬件联调)**:按下面的完整规则拆解。

### 拆分规则(模式B)

**拆分边界 = 执行主体切换点 或 关键验证点**,不是步骤数量。AI能连续独立完成、且中间没有高风险点的工作,合并为一个任务。只在以下情况新开任务节点:

- a) 切换到必须用户亲自处理的工作(参数调试、实机测试、采购硬件、物理组装、烧录部署等)
- b) 需要用户做关键决策才能继续
- c) 单个任务体量过大,AI一次性难以稳定完成
- d) **这一步是后续多个任务的地基**——一旦出错会连锁导致后面全错、难以定位,即使AI能力上可以连续往下写,也要在这里断开设一个验证checkpoint,不能让AI裸奔到最后才发现走偏

**每个任务标注执行者/checkpoint类型:**

- 🤖 **AI独立完成,风险低**:AI可以自己实现、自查、自测;通过后在"执行日志"记录结果,更新状态为✅,然后继续下一个任务,不用等待用户确认。适合普通代码改动、文档整理、脚本生成等。
- 🔍 **AI独立完成,但属于关键验证点**:AI能完成主要工作,但这一步是后续任务的地基或容易影响全局;完成后必须汇报验收结果,建议等用户确认无误后再继续,防止错误累积。适合架构方案、核心接口、数据结构、关键算法、硬件选型等。
- 🤝 **需要用户配合/验证**:AI能做一部分,但缺少用户提供的信息、实测反馈、账号/设备操作、主观选择或现场验证就不能闭环;AI完成能做的部分后停下来,明确请用户做什么、回传什么结果。
- 🧑 **用户亲自完成**:这一步AI无法代劳,只能给步骤、检查清单和注意事项;必须由用户实际执行。适合采购硬件、接线焊接、实机测试、烧录部署、现场安装、账号授权等。

**每个任务的描述要写到"AI可以直接动手执行"的详细程度**:开始前需要具备什么上下文、具体要实现什么功能及约束、用什么库/接口、预期产出物是什么(具体文件名/函数名/可观察效果)、验收标准怎么核查。如果某个实现细节AI自己也拿不准,要在任务里明确标出"此处需要先向用户确认XX,再动手",不要自己瞎猜带过去。

### 输出文档

复制 `references/task-doc-template.md` 的结构,生成/更新项目根目录下的 `开发任务拆解.md`。这份文档在整个项目周期里持续被读写,不会被替换——阶段三的所有记录也写在这一份文件里。

## 阶段三:执行阶段——维护同一份文档,不额外开progress文件

### 正常推进

- 🤖任务:做完自查/自测通过后,在文档的"执行日志"追加一条记录,更新"任务总览表"状态列为 ✅,直接继续下一个任务,不用每个都停下来汇报。
- 🔍任务:完成后在"执行日志"记录结果,但不要继续推进依赖它的后续任务;先按验收标准汇报,建议等待用户确认。
- 🤝任务:AI完成能做的部分后停下来,列清楚需要用户配合/验证的动作和回传信息。
- 🧑任务:不要假装已经完成;只给用户可执行步骤、注意事项和完成后的检查标准,等待用户实际执行并反馈。

执行日志只追加、不改写历史条目——哪怕后面发现某条记录已经过时,也不删除旧记录,而是追加一条新记录说明情况变了。这是保证"新会话能回溯"的关键:历史必须完整保留。

### 验证失败时怎么处理

1. 先在"异常记录"追加一条:任务编号、现象描述、发生时间。
2. 判断问题出在哪一层:
   - 这个任务本身实现有 bug → 修复后重做当前任务
   - 这个任务依赖的前置任务本身就有缺陷 → 回溯到那个前置任务重新检查/修改
   - 这个任务的方案设计本身就有问题(需求理解错了/技术选型不对)→ 回到阶段一,和用户重新讨论这部分方案
3. 把处理决定也追加进"异常记录",不要只记问题不记处理方式——这样以后遇到类似情况,读日志就知道之前怎么处理的,不用重新踩一遍坑。

### 方案/需求变更时怎么处理

不要重新生成整份文档,增量更新:

1. 顺着依赖关系,找出受影响的任务。
2. 作废的任务标记为"❌已作废"并保留(不删除),注明对应哪条变更记录。
3. 新增任务用新编号往后追加(如 T9、T10),不要打乱重排原有编号,避免依赖关系引用错乱。
4. 在"变更日志"追加一条记录:时间、改了什么、为什么改。
5. 更新受影响任务的依赖字段。

### 新会话/新agent接手项目时怎么读这份文档

1. 先看"任务总览表"的状态列,搞清楚整体进度。
2. 再看"执行日志"和"异常记录"最近几条,搞清楚最近发生了什么、有没有踩过坑、之前是怎么处理的。
3. 从状态为未完成的最早任务开始继续,不要因为一时看不懂就要求用户从头重新讲一遍方案——文档里的信息应该已经够用。

---

## 补充原则

- 语言简洁,不要为了体现"专业感"而堆砌形式化结构;能一句话说清楚的不要写三句。
- 验收标准尽量转化为可观察的客观标准(有具体输出/能跑通某测试/达到某指标),避免"效果好不好"这种主观描述。
- 任务详情里的"具体要做什么"要写到换一个新的AI会话/agent拿着这一条也能直接开始执行的详细程度。
- 如果用户觉得任务粒度偏细或偏粗,直接调整,不用纠结"标准流程应该怎样"。
