# 开发任务拆解.md 模板

> 这是「开发任务拆解.md」在模式B(中大型项目)下的完整结构。模式A(简单项目)只需要"复杂度评估" + "完成标准" + 少量检查点,跳过任务总览表和任务详情部分即可。
>
> 关键原则:**这一份文件贯穿项目全程**,不会被替换或另开一份progress文件。计划、实时状态、历史记录都在这一份里。执行日志/变更日志/异常记录三个区块只追加、不改写历史条目。

```markdown
# 开发任务拆解 —— [项目名]

## 复杂度评估
(简述判断依据,说明选择模式A还是模式B)

## 给AI的执行说明
- 按顺序执行下面的任务,若有依赖关系,严格按依赖顺序推进,不要跳过前置任务。
- 标🤖的任务:AI独立完成、风险低。自查/自测通过后直接继续下一个任务,不需要每完成一个就停下来汇报,但要在"执行日志"追加记录。
- 标🔍的任务:AI独立完成,但属于关键验证点。完成后必须停下来,按该任务的"验收标准"汇报结果,建议等待用户确认后再继续,防止错误累积。
- 标🤝的任务:需要用户配合/验证。AI先完成能做的部分,再停下来明确告诉用户需要做什么、提供什么信息、回传什么结果。
- 标🧑的任务:用户亲自完成。AI只能提供步骤、清单和注意事项,不能替用户完成;等待用户实际执行并反馈后再继续。
- 如果任务描述里的信息不足以让你直接动手(比如接口/参数/方案不明确),先提问澄清,不要自己假设关键实现细节。
- 如果是接手一个已经在进行中的项目:先看"任务总览表"状态列了解进度,再看"执行日志"和"异常记录"最近几条了解最近发生了什么,然后从最早的未完成任务继续,不要要求用户从头重新讲一遍背景。

## 任务总览表
(方便一眼看整体进度和依赖关系;状态列随执行实时更新,不需要保留历史——历史交给下面的执行日志)

| 编号 | 任务名称 | 执行者 | 依赖任务 | 状态 |
|---|---|---|---|---|
| T1 | ... | 🤖 | 无 | ⬜ |
| T2 | ...(核心逻辑/地基) | 🔍 | T1 | ⬜ |
| T3 | ... | 🧑 | T2 | ⬜ |

执行者说明:🤖 AI可连续推进 / 🔍 AI完成后需关键验收 / 🤝 需要用户配合或验证 / 🧑 用户亲自执行

状态说明:⬜未开始 / 🔄进行中 / ✅已完成 / ❌已作废(见变更日志)

## 任务详情(AI按此逐条执行)

### T1:任务名称
- **执行者**:🤖
- **依赖**:无
- **前置上下文**:开始前应具备什么(已有文件/已定义的接口/已有数据等)
- **具体要做什么**:详细到不用反复确认就能直接写代码——要实现的功能、约束条件、用什么库/接口、边界情况如何处理
- **预期产出物**:具体文件名/函数名/可观察到的效果
- **验收标准**:怎么判断做完了——具体测什么、看到什么结果算过

### T2:任务名称(🔍关键验证点)
- **执行者**:🔍
- **依赖**:T1
- (其余字段同上)
- **特别说明**:完成后请停下来汇报结果,等确认无误后再继续下一步

---

## 执行日志(只追加,不改写历史)
> 每完成一个任务追加一条。哪怕后面发现某条已过时,也不要删除或修改,追加新记录说明情况变化即可——这是保证新会话能完整回溯的关键。

- `[YYYY-MM-DD HH:MM] T1 完成 —— 做了什么 / 自测结果 / 是否有偏离原计划的调整`

## 变更日志(只追加)
> 方案或任务范围发生变更时记录。不重新生成整份文档,只做增量更新(见SKILL.md"方案/需求变更时怎么处理")。

- `[YYYY-MM-DD HH:MM] 变更内容 —— 原因 —— 受影响的任务编号(哪些作废、哪些新增)`

## 异常记录(只追加)
> 某个检查点验证失败时记录。必须同时记录"现象"和"处理决定",不能只记问题。

- `[YYYY-MM-DD HH:MM] T? 验证失败 —— 现象描述 —— 判断结果(重做当前任务 / 回溯到T? / 打回方案讨论)—— 后续处理结果`
```
